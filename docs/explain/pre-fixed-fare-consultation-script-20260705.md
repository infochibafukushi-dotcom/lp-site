## 1. 事前相談の目的

本事前相談は、関東運輸局に対して次のことをご確認いただくためのものです。

- 事前確定運賃認可申請に向けた、**独自開発の配車アプリ**の運用仕様が公示・様式の趣旨に沿っているか
- 旅客のルート選択・同意取得、乗務員への確定ルート提示、ルート変更時の扱い、精算・領収書表示、監査証跡の保存が、説明資料どおりに整理されているか
- 正式申請前に、不足している説明・別紙・運賃表の記載等があれば、事前に把握して修正する

**本相談は認可申請そのものではなく、「仕様の事前確認」を目的とします。** 認可の可否については、正式申請後の審査にお任せします。

## 5. 画面証跡 P1〜P5 の説明文

### P1：表紙・案件情報

「こちらは画面証跡資料の表紙です。本資料は、申請者本人による検証用予約データを用いた操作証跡です。見積番号 EST-20260705-3755、予約ID 202607050600、案件番号 260705-MAINS-0001、確定運賃 28,000円の案件について、旅客側・乗務員側・領収書までの一連の画面を記録しています。実運用前の検証データであることを冒頭でお伝えします。」

### P2：利用者の走行予定ルート選択画面

「旅客は、見積画面で2以上の走行予定ルート候補を確認し、その中から1つを選択します。時間優先・一般道優先・高速道路利用など、複数の算定条件に基づく候補を提示し、選択前は事前確定運賃として確定しません。独自開発のLP見積シミュレーター上で、候補一覧・推計距離・推計時間・運賃額を表示しています。」

### P3：旅客同意確認画面

「旅客がルートを選択したうえで、事前確定運賃の内容・走行予定ルート・運賃額・注意事項を確認し、同意操作を行います。同意日時と、見積内容のスナップショット（改ざん検知・照合用の証跡）を予約データとして保存します。技術的な詳細より、『同意した内容を後から照合できる』点を説明します。」

### P4：ドライバー用確定ルート確認画面

「乗務員のメーターアプリには、旅客が選択・同意したのと同一の確定ルートが表示されます。予約読取時に、保存されているスナップショットと照合し、確定運賃の整合を確認してから運行を開始します。旅客側と乗務員側で、同じ走行予定ルートを共有する設計です。」

### P5：領収書・レシート明細画面

「精算後の領収書には、『事前確定運賃』として運賃本体が表示されます。迎車料金・介助料・有料道路料金・実費等は、事前確定運賃とは区分して別明細で表示します。障害者割引等は精算時に乗務員が確認し、割引対象運賃・割引額・精算後金額を記録します。」

## 6. 想定される質問と短い回答

本節は事前相談で想定される質問への**短い回答例**です。詳細は Q&A資料（`pre-fixed-fare-qa-v1-candidate.pdf`）を参照してください。

### Q1. 旅客がルートをどう選ぶのか

「電子地図APIで2以上の走行予定ルート候補を取得し、見積画面に一覧表示します。旅客がその中から1つを選択した場合のみ、事前確定運賃として次の同意画面に進みます。選択前は確定しません。」

### Q2. 1ルートしか出ない場合はどうするのか

「ルート候補が1件しか取得できない場合、または旅客が複数ルートから選択した証跡を残せない場合は、事前確定運賃として確定せず、通常メーター運賃または個別確認による通常見積対応に切り替えます。」

### Q3. 有料道路利用の有無はどう扱うのか

「旅客が有料道路を利用するかどうかを選択できるようにし、その選択結果に基づく走行予定ルートで事前確定運賃を算定します。有料道路料金そのものは、事前確定運賃とは別に実費として区分表示します。」

### Q4. 旅客都合でルート変更した場合はどうするのか

「乗務員がメーターアプリ上で旅客都合によるルート変更・途中終了を操作します。当初の事前確定運賃による運送をその時点で終了し、変更後の区間は距離制運賃または新たな事前確定運賃による別運送として扱います。変更日時・理由・GPS位置・旅客同意・変更後区間の運賃処理を記録します。」

### Q5. 事故・通行止め・交通規制で迂回した場合はどうするのか

「交通事情・事故・通行止め・安全上の理由により乗務員判断で迂回する場合は、旅客の了承を得て運行します。旅客都合の変更ではないため、収受運賃は原則として当初の事前確定運賃額のままです。迂回理由・旅客了承・変更日時・GPS位置を記録します。」

### Q6. 運転者が確認する画面はあるのか

「あります。メーターアプリの確定ルート確認画面で、旅客が選択・同意した走行予定ルートと確定運賃を確認してから運行を開始します。画面証跡 P4 が該当画面です。」

### Q7. 領収書に事前確定運賃と表示されるのか

「はい。領収書・レシートに『事前確定運賃』として運賃本体を表示します。付帯料金・実費は別明細で区分します。画面証跡 P5 をご参照ください。」

### Q8. 割引と料金はどう区分しているのか

「見積・同意時点では、割引適用前の運賃本体（事前確定運賃額）を提示します。障害者割引等は精算時に乗務員が確認し、割引対象運賃・割引額・精算後金額を記録します。福祉タクシー券は運賃算定式には含めず、精算時の支払充当として処理します。」

### Q9. 保存期間は何年か

「運送終了日から3年間保存します。予約見積データ、旅客同意日時、選択ルート情報、スナップショット、運行ログ、ルート変更ログ、精算明細、領収書情報を対象とします。」

### Q10. 誰が証跡を閲覧できるのか

「閲覧・出力権限は、運行管理者、補助者、システム管理責任者に限定します。乗務員端末では、担当する当日運行または業務上必要な予約情報のみ閲覧可能とし、過去の監査証跡や他乗務員の運行記録にはアクセスできない権限制御を行います。」

### Q11. PDFやCSVで出力できるのか

「監査・苦情対応時には、予約ID・見積番号・運行日等をキーに対象データを確認できます。提出が必要な場合は、同意内容・選択ルート・運行記録・精算明細等を確認資料として整理し、PDF又はCSV等で出力できるよう、運用開始までに出力方法を明確化する予定です。本日お持ちの画面証跡・運用フロー資料が、その整理例です。」

### Q12. 実運用前の検証データか

「はい。画面証跡に使用している予約ID 202607050600 は、申請者本人による検証用予約データです。仕様確認のための操作証跡であり、一般旅客の実運行データではありません。正式運用開始前に、実機確認チェックリスト（別紙セット内）による目視確認も行う予定です。」

## 7. 説明時に言い切らない方がよい表現

### 使わない表現（NG）

| NG表現 | 理由 |
|--------|------|
| 「認可されます」「認可される見込みです」 | 認可判断は運輸局の審査による |
| 「完全準拠しています」「要件を満たしています」 | 自己断定にならないよう控える |
| 「実運用済みです」「既に運用しています」 | 検証データである事実と矛盾する |
| 「HMACで改ざんを完全に防止します」 | 技術詳細の押し出し・断定を避ける |
| 「GOと同じです」「他社と同様に問題ありません」 | 他事業者との比較断定を避ける |
| 「割引前後を見積時に必ず表示します」 | 現行は精算時確認が主。将来改善は別途 |

### 使う表現（OK）

| OK表現 | 用途 |
|--------|------|
| 「事前相談で仕様をご確認いただきたい」 | 相談の目的 |
| 「説明資料どおりの運用を想定しています」 | 仕様説明 |
| 「正式申請に向けて整理した候補資料です」 | 資料の位置づけ |
| 「独自開発の配車アプリです」 | システムの性質 |
| 「旅客が複数ルートから選択し、同意した証跡を保存します」 | コア仕様 |
| 「乗務員にも同一の確定ルートを表示します」 | 乗務員側の確認 |
| 「改ざん検知・照合用の証跡として保存します」 | snapshotHash の説明 |
| 「運送終了日から3年間保存する予定です」 | 保存期間 |
| 「運用開始までに出力方法を明確化します」 | 未確定事項の正直な説明 |

## 8. 事前相談後に確認する事項

相談終了後、次の点を記録・社内共有してください。

| 確認項目 | 記録内容 |
|----------|----------|
| 仕様上の指摘 | ルート選択・同意・ルート変更・精算のどの段階に関する指摘か |

| 申請書の修正要否 | 様式2の記載項目・営業区域・配車アプリ名称の修正 |
| 説明表現の修正要否 | 断定表現・技術説明の過不足 |
| 正式申請の手続き | 提出先・提出形式・事前相談の要否（再相談が必要か） |
| 運用開始前の TODO | 実機確認、PDF/CSV出力手順の明文化、権限設定の最終確認 |
| 次回アクション | 資料修正期限、正式申請予定日、追加質問への回答期限 |
